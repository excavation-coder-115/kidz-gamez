# Observability Research for Generated Applications

**Date:** 2026-03-21
**Status:** Research / Ideation

## Context

AppSheet provides three observability features that generated Exodus apps currently lack:

1. **Automation Monitor** — observe successes and failures on each automation and sub-task
2. **Audit Log** — log all sync, CRUD, and API operations
3. **Performance Profiler** — view specific sync details broken down by table, with virtual column calculation times highlighted

### Current State in Generated Apps

Generated apps have minimal observability:

- Pino logger in Fastify server (controlled via `LOG_LEVEL` env var)
- Basic try/catch in controllers with generic error responses
- `StepResult[]` tracking in `executeSteps()` (in-memory only, not persisted)
- Health check endpoints (`/api/health`, `/api/health/ready`)
- No distributed tracing, audit logging, query profiling, or automation monitoring

---

## Idea 1: Automation Execution Monitor

Mirror AppSheet's automation monitor by instrumenting the automation runtime with structured event logging and a query API.

### Why This Approach

The automation runtime already returns `StepResult[]` from `executeSteps()` with status tracking (`success | failed | skipped`). Extending this into a persistent audit trail is a natural fit — the data model exists, it just isn't stored.

### High-Level Design

**New schema table** generated by the Model Generator:

```typescript
// schema/automation_executions.ts
export const automationExecutions = pgTable('automation_executions', {
  id: uuid('id').defaultRandom().primaryKey(),
  botName: text('bot_name').notNull(),
  triggerType: text('trigger_type').notNull(),     // 'dataChange' | 'scheduled' | 'webhook'
  triggerTable: text('trigger_table'),
  triggerRowId: text('trigger_row_id'),
  status: text('status').notNull(),                // 'running' | 'success' | 'failed'
  startedAt: timestamp('started_at').defaultNow(),
  completedAt: timestamp('completed_at'),
  durationMs: integer('duration_ms'),
  steps: jsonb('steps'),                           // StepResult[] serialized
  error: text('error'),
});
```

**Wrap `executeSteps()` in the generated automation runtime** to persist results:

```typescript
// automation/scripts/automation-runtime.ts (generated)
async function executeAutomation(botName: string, triggerInfo: TriggerInfo, steps: StepDescriptor[]) {
  const execution = await db.insert(automationExecutions).values({
    botName,
    triggerType: triggerInfo.type,
    triggerTable: triggerInfo.table,
    triggerRowId: triggerInfo.rowId,
    status: 'running',
  }).returning();

  const start = performance.now();
  const results = await executeSteps(steps, triggerInfo.context, {
    onError: (stepName, error) => {
      fastify.log.error({ botName, stepName, error }, 'Automation step failed');
    },
  });

  const failed = results.some(r => r.status === 'failed');
  await db.update(automationExecutions)
    .set({
      status: failed ? 'failed' : 'success',
      completedAt: new Date(),
      durationMs: Math.round(performance.now() - start),
      steps: results,
      error: failed ? results.find(r => r.status === 'failed')?.output : null,
    })
    .where(eq(automationExecutions.id, execution[0].id));
}
```

**API endpoint for querying execution history:**

```typescript
// GET /api/_admin/automations
fastify.get('/api/_admin/automations', async (request, reply) => {
  const { botName, status, limit = 50 } = request.query;
  const rows = await db.select().from(automationExecutions)
    .where(and(
      botName ? eq(automationExecutions.botName, botName) : undefined,
      status ? eq(automationExecutions.status, status) : undefined,
    ))
    .orderBy(desc(automationExecutions.startedAt))
    .limit(limit);
  return { data: rows };
});
```

### Generators Affected

- **Model Generator** — emit `automation_executions` table schema
- **Automation Runtime Generator** — wrap `executeSteps()` with persistence
- **API Generator** — emit `/_admin/automations` query endpoint

### External Dependencies

None. Uses existing Drizzle ORM and Fastify infrastructure.

## Idea 2: Request Audit Log with Query Profiling

Combines AppSheet's audit log (CRUD/sync tracking) and performance profiler (query timing) into a single Fastify plugin generated into every app.

### Why This Approach

Fastify's lifecycle hooks (`onRequest`, `onResponse`) provide natural instrumentation points. Node.js `AsyncLocalStorage` allows correlating database queries back to the originating HTTP request without passing context explicitly — this gives us per-request query breakdowns similar to AppSheet's performance profiler.

### High-Level Design

**Fastify observability plugin** generated into `api/plugins/observability.ts`:

```typescript
import { FastifyPluginAsync } from 'fastify';

const observabilityPlugin: FastifyPluginAsync = async (fastify) => {
  // Assign request ID + start timer on every request
  fastify.addHook('onRequest', async (request) => {
    request.startTime = performance.now();
    request.queryLog = [];
  });

  // Log completed requests with timing + query breakdown
  fastify.addHook('onResponse', async (request, reply) => {
    const duration = performance.now() - request.startTime;
    const entry = {
      requestId: request.id,
      method: request.method,
      path: request.url,
      statusCode: reply.statusCode,
      userId: request.user?.id,
      durationMs: Math.round(duration),
      queryCount: request.queryLog.length,
      queryTimeMs: request.queryLog.reduce((sum, q) => sum + q.durationMs, 0),
      queries: request.queryLog,
      timestamp: new Date().toISOString(),
    };

    // Persistent audit log (async, non-blocking)
    db.insert(auditLog).values(entry).catch(err => fastify.log.error(err));

    // Structured log for external ingestion (DataDog, CloudWatch, etc.)
    fastify.log.info({ audit: entry },
      `${request.method} ${request.url} ${reply.statusCode} ${Math.round(duration)}ms`);
  });
};
```

**Drizzle query instrumentation** via `AsyncLocalStorage` in `database/connection.ts`:

```typescript
import { AsyncLocalStorage } from 'node:async_hooks';

const requestContext = new AsyncLocalStorage<{ queryLog: QueryEntry[] }>();

function instrumentedQuery(originalQuery: Function) {
  return async (...args: any[]) => {
    const start = performance.now();
    const result = await originalQuery(...args);
    const entry = {
      sql: args[0]?.sql || 'unknown',
      durationMs: Math.round(performance.now() - start),
      rowCount: Array.isArray(result) ? result.length : result?.rowCount ?? 0,
    };
    const ctx = requestContext.getStore();
    ctx?.queryLog.push(entry);
    return result;
  };
}
```

**Admin profiler endpoint:**

```typescript
// GET /api/_admin/profiler?minutes=30
fastify.get('/api/_admin/profiler', async (request) => {
  const { minutes = 30 } = request.query;
  const since = new Date(Date.now() - minutes * 60_000);
  const entries = await db.select().from(auditLog)
    .where(gte(auditLog.timestamp, since));

  return {
    totalRequests: entries.length,
    avgDurationMs: avg(entries, 'durationMs'),
    p95DurationMs: percentile(entries, 'durationMs', 95),
    slowestQueries: entries
      .flatMap(e => e.queries)
      .sort((a, b) => b.durationMs - a.durationMs)
      .slice(0, 20),
    byTable: groupBy(entries, e => extractTable(e.path)),
  };
});
```

### Generators Affected

- **API Generator** — emit observability plugin + register in `server.ts`
- **Model Generator** — emit `audit_log` table schema
- **Database Connection (in API Generator)** — add `AsyncLocalStorage` query wrapper

### External Dependencies

None. Uses `AsyncLocalStorage` (Node.js built-in), existing Drizzle ORM, and Fastify hooks.

## Recoverability: What Observability Enables for Free

Observability and recoverability share a fundamental dependency: **both require a structured event log**. The data you persist for debugging and monitoring is often the same data you need to retry, roll back, or reconstruct state after a failure. This means the observability approach you choose directly determines what recovery capabilities are available — or foreclosed — downstream.

### How Each Idea Maps to Recovery Primitives

#### Idea 1 (Automation Monitor) → Retry & Replay

The `automation_executions` table already captures the automation name, trigger context, per-step results, and the failure point. This is everything needed for three recovery patterns:

**Step-level retry.** Because `steps` is stored as a JSONB array of `StepResult[]` with individual statuses, the runtime can identify exactly which step failed and resume from that point rather than re-executing the entire automation. This avoids duplicate side effects (e.g., sending an email twice):

```typescript
// Possible extension to the generated runtime
async function retryExecution(executionId: string) {
  const execution = await db.select().from(automationExecutions)
    .where(eq(automationExecutions.id, executionId));

  const failedStepIndex = execution.steps.findIndex(s => s.status === 'failed');
  const remainingSteps = originalSteps.slice(failedStepIndex);

  await executeAutomation(execution.botName, {
    type: execution.triggerType,
    table: execution.triggerTable,
    rowId: execution.triggerRowId,
    context: execution.steps[failedStepIndex - 1]?.output, // carry forward last good state
  }, remainingSteps);
}
```

**Dead letter queue.** Failed executions sitting in the table with `status: 'failed'` are already a dead letter queue — they just need a consumer. A generated admin endpoint (`POST /_admin/automations/:id/retry`) plus a `retryCount` / `maxRetries` column turns passive failure records into an actionable retry system.

**Bulk replay.** For incident recovery (e.g., an external API was down for an hour), the execution history enables replaying all failed automations within a time window:

```typescript
// POST /_admin/automations/replay
// Body: { since: "2026-03-21T10:00:00Z", until: "2026-03-21T11:00:00Z", botName?: string }
```

**What the observability decision determines:** If step results are stored as an opaque blob or omitted entirely, step-level retry becomes impossible — you can only retry the full automation. Storing structured `StepResult[]` in JSONB (as proposed) preserves the granularity needed for surgical recovery.

#### Idea 2 (Audit Log + Profiler) → Undo, Rollback & Change History

The audit log captures every CRUD operation with method, path, user, and timing. Extending it slightly — to include the request body and the affected row's previous state — unlocks a different class of recovery:

**Per-record change history.** Filter the audit log by table + row ID and you have a complete changelog. This is a feature AppSheet exposes natively that generated apps currently lack:

```typescript
// GET /api/_admin/audit/changes?table=orders&rowId=abc-123
// Returns: [{ action: 'UPDATE', userId: '...', changedFields: {...}, previousValues: {...}, timestamp: '...' }]
```

**Point-in-time undo.** If the audit log stores `previousValues` alongside `changedFields`, an undo operation is a matter of applying the inverse:

```typescript
// POST /api/_admin/audit/:entryId/undo
async function undoAuditEntry(entryId: string) {
  const entry = await db.select().from(auditLog).where(eq(auditLog.id, entryId));
  await db.update(targetTable)
    .set(entry.previousValues)
    .where(eq(targetTable.id, entry.rowId));
}
```

**Batch rollback after incidents.** When a bad automation or a buggy release corrupts data across multiple rows, the audit log provides the "before" state for each affected record. This is the difference between "restore from last night's backup" and "surgically revert the 47 rows that were modified in the last 20 minutes."

**What the observability decision determines:** A minimal audit log (method + path + status code) enables performance analysis but not data recovery. Including `requestBody` and `previousValues` doubles the storage cost but unlocks undo/rollback. This is a design-time tradeoff that should be configurable via `GeneratorConfig`:

```typescript
// generator-config.ts
observability: {
  auditLog: true,
  auditLogRetainDays: 30,
  auditLogCaptureRequestBody: true,    // enables undo/replay
  auditLogCapturePreviousValues: true, // enables rollback — requires pre-read on mutations
}
```

#### OTel Traces → Targeted Recovery Triage

Distributed traces don't directly enable automated recovery, but they determine how quickly and accurately you can triage failures — which is the prerequisite for recovery.

Without traces, a failed automation surfaces as "status 500" and recovery means re-running the whole thing. With traces, you can see that the DB write succeeded but the downstream email step timed out — so the data is fine, only the notification needs resending. This distinction is the difference between "retry everything" and "retry only what failed," and it's only available if the observability layer captures the full request chain.

### Recoverability Cost by Observability Approach

| Recovery Capability | Minimum Observability Required | Storage Cost | Implementation Complexity |
|---|---|---|---|
| Retry full automation | Idea 1 (basic — status only) | Low | Low |
| Retry from failed step | Idea 1 (structured `StepResult[]` in JSONB) | Low | Low |
| Dead letter queue | Idea 1 + `retryCount` column | Low | Low |
| Bulk replay after incident | Idea 1 + time-range query | Low | Low |
| Per-record change history | Idea 2 + `requestBody` capture | Medium | Low |
| Single-record undo | Idea 2 + `previousValues` capture | Medium | Medium (pre-read on mutations) |
| Batch rollback | Idea 2 + `previousValues` capture | Medium | Medium |
| Targeted failure triage | OTel traces | Varies (external backend) | Low (auto-instrumentation) |

### Key Takeaway

The decisions made about observability granularity at generation time are effectively decisions about recoverability ceiling. Storing structured step results instead of a pass/fail flag, capturing previous values instead of just the new state, and tracing full request chains instead of logging individual endpoints — each of these is a marginal cost increase in the observability layer that unlocks a qualitatively different tier of recovery capability. The recommendation is to default to the richer capture (with configurable retention) so that recovery options are available when needed, rather than requiring a code regeneration to add them after an incident.

## External Platform Integrations

### Grafana Stack (Backend Observability)

The Grafana open-source ecosystem is a strong fit for generated apps, especially self-hosted deployments where SaaS pricing (e.g., DataDog) is prohibitive.

**Components:**

- **Grafana Loki** — log aggregation. Pino logs pipe directly via `pino-loki` transport, replacing the need for a custom `audit_log` table if the user prefers external log storage.
- **Grafana Tempo** — distributed tracing. Works with OpenTelemetry (see below).
- **Prometheus** — metrics collection. Fastify has a `fastify-metrics` plugin that exposes request duration histograms, error rate counters, and custom gauges at `/metrics` with near-zero setup.
- **Grafana Dashboards** — we could generate pre-built dashboard JSON files that visualize automation success/failure rates, API latency by table, slow queries, etc.

**Why Grafana over DataDog for generated apps:**

| | DataDog | Grafana Stack |
|---|---|---|
| Cost | Per-host pricing, expensive at scale | Free / self-hosted |
| Setup | SaaS, just add API key | Requires running Loki/Tempo/Prometheus |
| Vendor lock-in | High (`dd-trace` is proprietary) | Low (OpenTelemetry standard) |
| Best for | Teams already paying for DataDog | Self-hosted apps, cost-conscious users |

**OpenTelemetry as the vendor-neutral layer:**

Rather than generating vendor-specific instrumentation, we could generate OpenTelemetry (OTel) auto-instrumentation. This works with Grafana Tempo, DataDog, New Relic, or any OTel-compatible backend — one generated integration, multiple backends:

```typescript
// generated in server.ts — vendor-neutral
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';

const sdk = new NodeSDK({
  serviceName: 'my-generated-app',
  instrumentations: [getNodeAutoInstrumentations()],
  // Auto-detects OTEL_EXPORTER_* env vars — works with Grafana Tempo, DataDog, etc.
});
sdk.start();
```

This auto-instruments Fastify requests, database queries (pg driver), and HTTP calls from automations — no manual wrapping needed. The user sets `OTEL_EXPORTER_OTLP_ENDPOINT` to point at their preferred backend.

### PostHog (Frontend Observability)

PostHog covers the client-side observability gap that neither Idea 1 nor Idea 2 address — understanding what users are doing in the generated React app.

**Relevant features:**

- **Session Replays** — video-like recordings of user sessions, useful for debugging UI issues in generated views (Table, Form, Detail, etc.)
- **Product Analytics** — track custom events like CRUD actions, form submissions, navigation patterns, and error occurrences
- **Error Tracking** — capture uncaught React errors with stack traces and session context
- **Feature Flags** — could be used to gate observability features themselves (e.g., enable profiler for admin users only)

**Integration in generated apps:**

```typescript
// generated in client/src/main.tsx
import posthog from 'posthog-js';

if (import.meta.env.VITE_POSTHOG_KEY) {
  posthog.init(import.meta.env.VITE_POSTHOG_KEY, {
    api_host: import.meta.env.VITE_POSTHOG_HOST || 'https://us.i.posthog.com',
    autocapture: true,          // auto-tracks clicks, page views, form submissions
    session_recording: { recordCrossDomainIFrames: true },
  });
}
```

**Generated event tracking in hooks:**

```typescript
// generated in client/src/hooks/useTable.ts
const useTable = (tableName: string) => {
  const create = async (data: Record<string, unknown>) => {
    const result = await api.post(`/api/${tableName}`, data);
    posthog?.capture('record_created', { table: tableName });
    return result;
  };

  const remove = async (id: string) => {
    await api.delete(`/api/${tableName}/${id}`);
    posthog?.capture('record_deleted', { table: tableName, id });
  };
  // ...
};
```

**Generated React error boundary:**

```typescript
// generated in client/src/components/ErrorBoundary.tsx
class ErrorBoundary extends React.Component {
  componentDidCatch(error: Error, info: React.ErrorInfo) {
    posthog?.capture('$exception', {
      $exception_message: error.message,
      $exception_stack_trace_raw: error.stack,
      $exception_type: error.name,
      component_stack: info.componentStack,
    });
  }
  // ...
}
```

**Why PostHog over alternatives (Sentry, LogRocket, FullStory):**

- Open-source and self-hostable (aligns with Grafana philosophy)
- Combines analytics + session replay + error tracking in one tool (fewer integrations to generate)
- Generous free tier for SaaS usage
- Simple `autocapture` mode means generated apps get useful data with minimal custom instrumentation

### Activation Strategy

Both integrations would be **opt-in via environment variables** — generated apps work standalone with Ideas 1 & 2, but light up with richer observability when keys are provided:

- **Infrastructure Generator** — optionally include Grafana/Loki/Tempo/Prometheus in `docker-compose.yml`
- **Config Generator** — add env var entries to `.env.example`

## Performance Impact Analysis

Every observability and recoverability feature adds overhead — extra writes, memory allocations, network calls, or CPU cycles on the critical path. The right question isn't whether there's a cost, but whether the cost lands on the hot path and whether it's proportional to the value.

### Per-Feature Overhead Breakdown

#### Idea 1: Automation Execution Monitor

**Hot path impact: Low.**

Automations are already I/O-bound (they send emails, call APIs, write rows). The two additional database writes per execution — one `INSERT` at start, one `UPDATE` at completion — are negligible relative to the automation's own work:

| Operation | Added Latency | Frequency | On Critical Path? |
|---|---|---|---|
| `INSERT` execution record at start | ~1-3ms | Once per automation run | Yes, but automation hasn't started yet |
| `UPDATE` with results at completion | ~2-5ms | Once per automation run | No — automation is already done |
| JSONB serialization of `StepResult[]` | <1ms | Once per automation run | No — happens during completion write |

**Storage growth:** Each execution record is roughly 1-4 KB (depending on step count). At 1,000 automations/day, that's ~4 MB/day or ~120 MB/month before retention cleanup. Manageable without indexing concerns for typical generated apps.

**With retry/replay extensions:** Retry adds one more `INSERT` per retry attempt. Bulk replay could spike write volume temporarily, but since it's admin-initiated and rate-limitable, it won't surprise the database.

#### Idea 2: Audit Log + Query Profiler

**Hot path impact: Medium — but tunable.**

The Fastify `onRequest`/`onResponse` hooks add minimal overhead (object allocation + timestamp). The heavier costs come from what you choose to capture:

| Operation | Added Latency | Frequency | On Critical Path? |
|---|---|---|---|
| `onRequest` hook (start timer, init queryLog) | <0.1ms | Every request | Yes |
| `onResponse` hook (compute duration, build entry) | <0.5ms | Every request | Yes (but response already sent to client if using `onSend`) |
| Async `INSERT` into `audit_log` | ~2-5ms | Every request | **No — fire-and-forget** |
| `AsyncLocalStorage` context propagation | ~0.05ms per DB call | Every DB query | Yes |
| Query timing wrapper | <0.1ms per query | Every DB query | Yes |

**The `previousValues` capture for recoverability is the most expensive addition.** It requires a `SELECT` before every `UPDATE`/`DELETE` to read the current state. This effectively doubles the query count for mutation operations:

| Mutation Type | Without `previousValues` | With `previousValues` |
|---|---|---|
| `UPDATE` | 1 query | 2 queries (SELECT + UPDATE) |
| `DELETE` | 1 query | 2 queries (SELECT + DELETE) |
| `INSERT` | 1 query | 1 query (no previous state) |

For read-heavy apps (typical of AppSheet migrations), this cost is minor. For write-heavy apps with high mutation throughput, this could be significant. Hence the recommendation to make it configurable:

```typescript
// When auditLogCapturePreviousValues is false, skip the pre-read
if (config.observability.auditLogCapturePreviousValues) {
  const previous = await db.select().from(table).where(eq(table.id, id));
  auditEntry.previousValues = previous[0];
}
```

**Storage growth:** Audit log entries are larger than execution records — roughly 2-8 KB each depending on request/response body capture. At 10,000 requests/day, that's ~80 MB/day. Retention cleanup (the 30-day prune mentioned in Implementation Notes) is essential, and indexing on `timestamp` + `path` is needed to keep query performance stable as the table grows.

#### OTel Auto-Instrumentation

**Hot path impact: Low-Medium.**

The `@opentelemetry/auto-instrumentations-node` package monkey-patches core modules (HTTP, `pg` driver, etc.) to inject trace context. Benchmarks from the OTel community show:

| Metric | Typical Overhead |
|---|---|
| Request latency increase | 1-5% (dominated by context propagation) |
| Memory increase | 10-30 MB baseline for the SDK + span buffer |
| CPU increase | 1-3% for span creation/serialization |
| Network (export) | Batch export every 5s, ~1-10 KB per batch depending on traffic |

The key variable is **exporter configuration**. The default `BatchSpanProcessor` buffers spans and exports in batches, keeping per-request overhead low. Switching to `SimpleSpanProcessor` (synchronous export) would put network latency on the hot path — this should never be the generated default.

**When OTel is not configured** (no `OTEL_EXPORTER_OTLP_ENDPOINT` set), the SDK initializes in no-op mode. There is still a small cost from the monkey-patching, but span creation becomes a no-op. If even this is unacceptable, the generated code should guard the `sdk.start()` call behind the env var check so the patches are never applied.

#### PostHog (Frontend)

**Hot path impact: Negligible on server, Low on client.**

PostHog runs entirely client-side and batches events for async upload. The main costs are:

| Metric | Impact |
|---|---|
| Bundle size increase | ~20-25 KB gzipped (posthog-js) |
| Session recording bandwidth | 10-50 KB/min per active session (compressed DOM snapshots) |
| JavaScript main thread | <1ms per captured event; session recording uses MutationObserver (off main thread for DOM diffing) |
| API calls to PostHog | Batched every 30s by default |

Session recording is the heaviest feature. For apps with many concurrent users, the bandwidth to PostHog's servers (or a self-hosted instance) can add up. This is already opt-in via the `VITE_POSTHOG_KEY` env var, and session recording can be independently disabled if needed.

### Cumulative Impact: All Features Enabled

For a generated app running all observability features simultaneously:

| Metric | Estimated Total Overhead |
|---|---|
| Per-request latency (read) | +1-5ms (OTel context + audit hook + query timing) |
| Per-request latency (write, with `previousValues`) | +5-15ms (above + pre-read SELECT) |
| Memory baseline | +15-40 MB (OTel SDK + span buffer + Pino transport buffers) |
| Database writes | +1 INSERT per request (audit) + 2 per automation (execution monitor) |
| Storage growth | ~100-200 MB/month at moderate traffic (before retention prune) |
| Client bundle size | +20-25 KB gzipped (PostHog) |

These numbers are well within acceptable bounds for the class of application being generated (internal tools, line-of-business apps migrated from AppSheet). The apps most likely to hit performance limits are high-write-throughput automation-heavy deployments — for these, the `GeneratorConfig` flags allow selectively disabling the costliest features (`previousValues` capture, session recording) without sacrificing core observability.

### Mitigation Strategies for Generated Code

1. **Async audit writes.** The `db.insert(auditLog)` call in the `onResponse` hook should always be fire-and-forget (`.catch(err => log.error(err))`). Never `await` it on the response path.
2. **Batch OTel export.** Always generate `BatchSpanProcessor`, never `SimpleSpanProcessor`. Set `scheduledDelayMillis` to 5000ms (the default).
3. **Audit log indexing.** Generate a composite index on `(timestamp, path)` and a partial index on `status_code >= 400` for error queries.
4. **Retention as a first-class generated feature.** Generate a scheduled job (or pg_cron statement) that prunes `automation_executions` and `audit_log` rows older than the configured retention window. Without this, storage growth is unbounded.
5. **Conditional `previousValues` capture.** Gate behind config flag. For apps that don't need undo/rollback, skip the pre-read entirely.
6. **No-op guards for OTel.** Wrap `sdk.start()` in `if (process.env.OTEL_EXPORTER_OTLP_ENDPOINT)` so the monkey-patching is never applied when tracing isn't configured.
7. **PostHog sampling.** For high-traffic generated apps, generate a `session_recording: { sampleRate: 0.1 }` config to record only 10% of sessions by default.

## Implementation Notes

- Both ideas are **additive** — they layer instrumentation on top of existing generated code without changing its structure.
- Could be gated behind a `GeneratorConfig` flag (e.g., `observability: true`) so apps can opt in.
- The `/_admin/*` endpoints should be protected with an admin-role check in generated auth middleware.
- Audit log tables should have a retention/cleanup strategy (e.g., generated cron job to prune entries older than 30 days).
- Both ideas are **independent and complementary** — they can be implemented together or separately.
